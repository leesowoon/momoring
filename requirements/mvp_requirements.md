# Momoring MVP 요구사항 정의서 (STS 캐릭터 챗봇)

## 1. 문서 목적
본 문서는 Momoring 서비스를 MVP 단계까지 구현하기 위한 요구사항을 정의합니다.
대상 범위는 **음성 기반 대화(STT→LLM→TTS) + 캐릭터 표시 + 안전 가드레일**입니다.

---

## 2. 제품 목표
- 아이/청소년 사용자가 Momoring과 자연스럽게 음성 대화를 한다.
- Momoring은 친근한 말투로 연령 맞춤 설명을 제공한다.
- 미성년자 대상 서비스로서 안전하고 신뢰 가능한 대화 환경을 제공한다.

---

## 3. MVP 범위

### 3.1 포함 범위 (In Scope)
1. 실시간에 준하는 음성 대화
   - 사용자 음성 입력(STT)
   - LLM 응답 생성
   - 음성 합성(TTS) 출력
2. 캐릭터 대화 UI
   - 상태: 대기/듣는 중/생각 중/말하는 중
   - TTS 출력과 립싱크 연동
3. 연령 맞춤 응답
   - 연령 구간별 난이도/어휘/설명 길이 조정
4. 안전 가드레일
   - 입력/출력 필터링
   - 민감 대화 대응 템플릿
5. 운영 최소 기능
   - 세션 로그, 안전 이벤트 로그, 사용자 피드백 수집

### 3.2 제외 범위 (Out of Scope)
- 멀티 캐릭터
- 고급 개인화(장기 기억 학습)
- 결제/구독
- 외부 학습 플랫폼 정식 연동

---

## 4. 사용자/시나리오 요구사항

### 4.1 주요 사용자
- 7~15세 아동/청소년
- 보호자(파일럿 동의 및 안전 신뢰 확인)

### 4.2 핵심 시나리오
1. 사용자가 마이크 버튼을 누르고 질문한다.
2. Momoring이 질문을 이해하고 다정한 톤으로 대답한다.
3. Momoring 캐릭터가 말할 때 입모양/상태가 자연스럽게 전환된다.
4. 위험/민감 발화가 발생하면 안전 정책 기반 응답으로 전환한다.

---

## 5. 기능 요구사항 (Functional Requirements)

### FR-01 STT 입력
- 시스템은 사용자 음성을 텍스트로 변환해야 한다.
- 발화 종료를 감지해야 한다(VAD 또는 push-to-talk 종료 이벤트).

### FR-02 대화 오케스트레이션
- 시스템은 변환된 사용자 발화를 LLM에 전달해야 한다.
- 시스템은 연령 정책을 적용하여 응답을 생성해야 한다.

### FR-03 TTS 출력
- 시스템은 최종 응답을 음성으로 합성해 재생해야 한다.
- 시스템은 캐릭터 발화 상태를 TTS 재생 구간과 동기화해야 한다.

### FR-04 캐릭터 상태 표시
- 시스템은 최소 4개 상태(대기/듣기/생각/말하기)를 시각적으로 표시해야 한다.
- 상태 전이는 대화 파이프라인 이벤트와 연결되어야 한다.

### FR-05 안전 필터
- 시스템은 사용자 입력과 모델 출력을 모두 안전 필터링해야 한다.
- 고위험 시나리오에서 사전 정의된 안전 응답을 우선 적용해야 한다.

### FR-06 로깅/모니터링
- 시스템은 세션별 대화 턴, 지연시간, 오류, 안전 이벤트를 기록해야 한다.

### FR-07 피드백 수집
- 사용자는 각 응답에 대해 긍정/부정 피드백을 남길 수 있어야 한다.

---

## 6. 비기능 요구사항 (Non-Functional Requirements)

### NFR-01 성능
- 응답 지연시간 P95: 2.5초 이내를 목표로 한다.

### NFR-02 안정성
- 네트워크/STT/TTS 오류 시 fallback 응답을 제공해야 한다.
- 세션 실패율 3% 이하를 목표로 한다.

### NFR-03 보안/개인정보
- 아동 개인정보는 최소 수집 원칙을 따른다.
- 로그는 서비스 운영 목적 범위 내에서만 저장한다.

### NFR-04 안전성
- 유해 출력 미노출률 99.5% 이상 목표
- 고위험 발화 감지 재현율 95% 이상 목표

---

## 7. API 요구사항 (초안)
- `POST /v1/sts/session/start`
- `POST /v1/sts/stream`
- `POST /v1/sts/respond`
- `POST /v1/tts/speak`
- `POST /v1/safety/check`

각 API는 요청/응답 스키마와 에러코드를 OpenAPI 문서로 별도 관리한다.

---

## 8. 데이터 요구사항
- 세션: 사용자 연령대, 시작/종료 시각, 세션 상태
- 턴: 사용자 발화 텍스트, 모델 응답 텍스트, 처리 지연시간
- 안전 이벤트: 유형, 심각도, 조치 결과

---

## 9. 수용 기준 (Acceptance Criteria)
MVP는 아래 조건을 만족하면 완료로 판단한다.
1. 음성 입력→응답 출력의 엔드투엔드 대화가 동작한다.
2. 캐릭터 상태 전환 및 립싱크가 정상 동작한다.
3. 안전 필터가 입력/출력 모두에서 작동한다.
4. 핵심 지표(지연시간/안전/피드백)를 수집할 수 있다.
5. 내부 파일럿 운영이 가능한 수준으로 안정화되어 있다.

## 12. AI 선정 요구사항
- MVP는 LLM/STT/TTS에 대해 Primary/Fallback 이중 경로를 가져야 한다.
- LLM Primary는 GPT-5.4 우선, 미가용/지연 초과 시 Claude 자동 전환을 기본 정책으로 한다.
- 모델 선택 기준은 안전성, 한국어 품질, 지연시간, 비용을 포함해야 한다.
- 모델/버전 변경 시 단계적 배포와 롤백 전략을 적용해야 한다.
- 상세 기준은 `requirements/mvp_technical_design.md` 19장을 따른다.
